本仓库遵循 **“多代理博弈与自动化硬门禁”** 的开发流程。通过将“生产（实现）”与“审计（测试）”在模型逻辑上分离，实现高可靠、低心智负担的自动化开发。

# 角色分工 (Model Roles)

| **代理人格**             | **核心模型建议**   | **职责描述**                                              |
| ------------------------ | ------------------ | --------------------------------------------------------- |
| **厂长 (Human)**         |                    | 定义核心需求，进行最终决策与 Git 提交。                   |
| **架构代理 (Architect)** | **Opus 4.6**       | 逻辑分析、技术选型、编写 `implementation-plan.md`。       |
| **视觉/UX 代理 (UX)**    | **Gemini-3.1-Pro** | UI 组件开发、CSS 样式、利用 `browser-use` 进行 E2E 验证。 |
| **审计代理 (QA)**        | **GPT-5.3-Codex**  | 进行 Code Review、确保 0 Issue。                          |

# 五阶段标准流水线 (The Pipeline)

## Phase 1: 策略规划 (Strategy)

- **输入**：厂长的原始需求。
- **输出**：`implementation-plan.md`。
- **动作**：分析现有代码逻辑，明确变更范围，禁止直接编码。

## Phase 2: 契约定义 (Contracting / TDD)

- **输入**：技术方案。
- **输出**：`*.test.ts` (Vitest/Jest)。
- **动作**：审计代理编写测试。运行 `npm test` 确认失败（Red），确立“交付契约”。

## Phase 3: 自动化生产 (Implementation)

- **输入**：方案 + 测试用例。
- **输出**：业务代码 (Node.js/TS)。
- **动作**：逻辑/视觉代理并行作业。目标是使测试变绿（Green），且符合 Lint 规范。

## Phase 4: 物理门禁 (Engineering Gate)

- **必经点**：
  1. `npm run lint`：静态代码审计通过。
  2. `npm test`：所有业务逻辑单元测试 100% 通过。
  3. `E2E Verify`：核心链路通过浏览器自动化脚本验证。

## Phase 5: 归档交付 (Shipment)

- **动作**：AI 汇总变更日志。
- **终态**：厂长核对“回车”，执行 `git commit`。

# 项目环境约束 (Node.js)

- **模块化**：严格使用 ESM (`import/export`)。
- **安全性**：禁止硬编码，必须使用 `.env` 管理环境变量。
- **健壮性**：异步逻辑必须包含错误边界（Error Boundary）或 `try-catch`。
- **可测试性**：前端组件必须包含 `data-testid` 以便 Agent 抓取。

# 厂长禁令 (Non-Negotiables)

1. **禁止跳步**：严禁在没有测试用例的情况下编写业务代码。
2. **禁止寒暄**：AI 汇报应保持极简，优先使用表格或 Markdown 列表。
3. **纠错优先**：若需求不合理，AI 必须提出反驳建议而非盲目执行。
4. **数据沉淀**：每次解决复杂 Bug 后，必须将经验同步至 `claude.md` 的“资产库”部分。
